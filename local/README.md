# local — Quick mode

Two apps and dynamoip running in Docker with `.local` hostnames and mkcert HTTPS.
No Cloudflare account needed. Works on macOS, Linux, and Windows.

```
https://inventory.local  →  Docker container (port 3001)
https://dashboard.local  →  Docker container (port 6000)
```

---

## How it works

dynamoip's Quick mode uses `.local` hostnames and certificates from a local CA (generated by mkcert). The challenge with Docker is that each container has its own network namespace — the normal approach of running dynamoip directly on the host doesn't apply. This example solves it in two parts:

**Host setup (once):**
- `mkcert -install` adds the mkcert CA to your system keychain. Your browser trusts any cert signed by this CA.
- `/etc/hosts` entries point `inventory.local` and `dashboard.local` to `127.0.0.1`.
- The path to your mkcert CA directory (`CAROOT`) is saved to `.env`.

**Docker (ongoing):**
- The host's mkcert CA directory is mounted read-only into the dynamoip container.
- The container's `CAROOT` env var tells mkcert where to find that CA.
- `mkcert -install` runs inside the container, registering the host CA in the container's trust store.
- dynamoip issues TLS certificates signed by the host CA — your browser already trusts them.
- socat runs inside the dynamoip container to bridge `localhost:PORT` → Docker service names, since containers can't reach each other via localhost.
- Ports 80 and 443 are mapped from the host into the dynamoip container (like Pro mode).

```
Your machine
  /etc/hosts: inventory.local → 127.0.0.1
              dashboard.local → 127.0.0.1

  Port 80/443 → dynamoip container
                  socat: localhost:3001 → inventory container
                  socat: localhost:6000 → dashboard container
                  mkcert CA: mounted from host (same CA browser already trusts)
                  dynamoip: issues certs, proxies HTTPS by hostname
```

## Setup

### Prerequisites

- [Docker Desktop](https://docs.docker.com/get-docker/) (macOS/Windows) or Docker Engine (Linux)
- Node.js >= 14
- mkcert:
  - macOS: `brew install mkcert`
  - Linux: see [mkcert installation](https://github.com/FiloSottile/mkcert#linux)
  - Windows: `choco install mkcert`

---

### Step 1 — Run host setup

**macOS / Linux:**
```bash
sudo node setup-hosts.js
```

**Windows** (run terminal as Administrator):
```cmd
node setup-hosts.js
```

This script:
1. Checks mkcert is installed
2. Runs `mkcert -install` — adds the CA to your system keychain (macOS shows a password prompt)
3. Writes `CAROOT` to `.env` — the path Docker will mount into the container
4. Adds `inventory.local` and `dashboard.local` to `/etc/hosts`

You only need to run this once. It is safe to re-run.

---

### Step 2 — Start everything

```bash
docker compose up --build
```

Docker will:
1. Build the inventory, dashboard, and dynamoip images
2. Start inventory and dashboard, wait for their healthchecks to pass
3. Start dynamoip — it installs the host CA inside the container, starts socat forwarders, then begins proxying

---

### Step 3 — Open in your browser

```
https://inventory.local
https://dashboard.local
```

No certificate warnings. The certs are signed by the mkcert CA that is already in your keychain.

## Other devices on the same network

Other devices need to install the mkcert CA cert once. After running `setup-hosts.js`, the CA cert is at:

```
$(mkcert -CAROOT)/rootCA.crt
```

Transfer this file to the other device and install it:

| Platform | How to install |
|---|---|
| macOS | Double-click the file → Keychain Access → mark as **Always Trust** |
| Linux | `sudo cp rootCA.crt /usr/local/share/ca-certificates/ && sudo update-ca-certificates` |
| Windows | Double-click → Install Certificate → Local Machine → Trusted Root Certification Authorities |
| iOS | AirDrop the file → Settings → General → VPN & Device Management → install the profile → Settings → General → About → Certificate Trust → enable |
| Android | Settings → Security → Encryption & Credentials → Install a certificate |

Each other device also needs `/etc/hosts` entries (or use your router's DNS):
```
127.0.0.1 inventory.local dashboard.local
```
Replace `127.0.0.1` with your machine's LAN IP on those devices.

## Troubleshooting

**`CAROOT is not set` error when running `docker compose up`**
→ Run `setup-hosts.js` first. It writes `CAROOT` to `.env`.

**Certificate warning in browser**
→ The mkcert CA is not trusted on this device. Install `rootCA.crt` (see above).

**`inventory.local` / `dashboard.local` not resolving**
→ Check `/etc/hosts` has the entries. Re-run `sudo node setup-hosts.js` if missing.

**Port 80 or 443 already in use**
→ Another process (nginx, Apache, another Docker container) is bound to those ports. Stop it first.

## For AI / LLM context

**Why this approach exists:**
Quick mode normally runs dynamoip directly on the host with `sudo`, which gives it direct access to the network interface for mDNS and the system keychain for mkcert. Docker isolates both. This example replicates the same end result by splitting responsibilities: the host handles CA trust and hostname resolution (`/etc/hosts`), Docker handles everything else.

**Key design decisions:**
- `CAROOT` is written to `.env` by `setup-hosts.js` and mounted as a read-only volume. This avoids hardcoding a path (it differs per user and OS) and keeps the docker-compose.yml portable.
- `CAROOT` env var is set explicitly in `docker-compose.yml` (pointing to the container-side mount path) so mkcert inside the container uses the mounted CA, not a new one.
- socat is included in the dynamoip container (same as Pro and Max modes) because containers cannot reach each other via localhost.
- No `network_mode: host` — it doesn't work on macOS/Windows Docker Desktop and is unnecessary here since `/etc/hosts` + port mapping achieves the same result.
- `mkcert -install` in the entrypoint is idempotent and registers the mounted host CA in the container's system trust store.
